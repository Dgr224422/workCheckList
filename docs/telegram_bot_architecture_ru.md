# ТЗ и архитектура Telegram-бота для кинотеатра

## 1) Цели проекта
Создать Telegram-бота на **aiogram** для автоматизации:
- учёта сертификатов (по коду и QR);
- учёта расхода попкорна (3 объёма вёдер);
- учёта школьных/классных групп (рейтинг, посещения, фото-подтверждения);
- чек-листов сотрудников с напоминаниями;
- построения графика на месяц администратором.

Ключевое требование: разделение данных по независимым функциональным модулям (логически отдельные БД/схемы), централизованное логирование, ролевой доступ, защита от атак.

---

## 2) Рекомендуемый стек

### Ядро
- Python 3.12+
- `aiogram` (v3)
- `asyncio`

### База данных
Вариант A (рекомендуется для начала):
- PostgreSQL как СУБД
- Логическое разделение на **отдельные схемы**:
  - `certificates`
  - `inventory`
  - `classes`
  - `checklists`
  - `schedule`
  - `auth`
  - `audit`

Вариант B (строго «несвязные БД»):
- 4–6 отдельных БД PostgreSQL (по модулю), общий сервисный слой доступа.

### ORM / доступ
- SQLAlchemy 2.x (async)
- Alembic для миграций

### Файлы и фото
- S3-совместимое хранилище (MinIO/S3) или локальное файловое хранилище
- Pillow/OpenCV для валидации изображений
- pyzbar / zxing-cpp-python для QR-распознавания

### Очереди / планировщик
- APScheduler или Celery + Redis (для напоминаний и фоновых задач)

### Наблюдаемость и логирование
- `structlog` или стандартный `logging` + JSON формат
- Sentry для ошибок

## 3) Роли и права

### Роли
1. **system_admin**
   - Полный доступ ко всем разделам
   - Назначает/удаляет администраторов
   - Может редактировать любые записи
2. **admin**
   - Управляет чек-листами, графиком, отчётами
   - Может редактировать все прикладные записи
3. **worker**
   - Работает с ежедневными операциями (чек-листы, расход, поиск сертификатов)

### Идентификация
- В `.env` хранится `SYSTEM_ADMIN_TELEGRAM_ID`
- Таблица пользователей с ролью и аудитом изменений

## 5) Модуль «Сертификаты»

### Функции
1. Поиск по QR (с фото)
2. Поиск по полному текстовому коду
3. Поиск по части кода (autocomplete)
4. Проверка статуса сертификата (активен/погашен/истёк)
5. История операций

### Логика
- При получении фото бот пытается распознать QR.
- Извлекает код и ищет в БД.
- Если QR не распознан, предлагает ручной ввод части кода.
- Для частичного кода выдаёт список совпадений (limit, например 10) с кнопками выбора.

### Пример таблиц
- `certificates.certificate`
  - `id`, `code` (unique), `qr_payload`, `status`, `nominal`, `expires_at`, `created_at`
- `certificates.certificate_event`
  - `id`, `certificate_id`, `event_type`, `actor_user_id`, `created_at`, `meta`

## 6) Модуль «Расход попкорна»

Учёт по трём типам:
- 1.5 л (в рукаве 40 вёдер)
- 3.0 л (в рукаве 40 вёдер)
- 6.0 л (в рукаве 25 вёдер)

### Ежедневный сценарий ввода
Для **каждого объёма отдельно** сотрудник вводит:
1. Сколько рукавов принесли со склада
2. Продано по кассе (шт)
3. Остаток в ТЗ (шт)
4. Фото отчёта (обязательно)

Система знает/фиксирует:
- Утренний остаток на складе
- Вчерашний итоговый остаток (конец дня)

### Формулы
Обозначения:
- `S_morning` — остаток на складе утром
- `R` — рукавов принесли
- `K` — коэффициент вёдер в рукаве (40/40/25)
- `Sold_cash` — продано по кассе
- `TZ_end` — остаток в ТЗ в конце дня
- `S_after_issue = S_morning - (R * K)` — склад после выдачи в ТЗ
- `End_day = S_after_issue + TZ_end`
- `Delta_prev = Prev_end_day - End_day` — фактическое снижение запаса

Проверка:
- Ожидаемое проданное: `Sold_cash`
- Фактическое по остатку: `Delta_prev`
- `Diff = Sold_cash - Delta_prev`

Если `Diff != 0`, бот сообщает расхождение: 
- «по кассе больше на N» или «по кассе меньше на N».

### Хранение и ротация данных
- Хранение детальных дневных записей: **45 дней**
- Ежедневный cron удаляет записи старше 45 дней (по дням, от самых старых)
- Можно сохранять агрегаты (необязательно)

### Отчёты
- За конкретную дату
- За последние 7/14/28 дней
- Форматы:
  - текстовый свод
  - свод + ссылки на фото/файлы

### Таблицы
- `inventory.popcorn_daily`
  - `id`, `date`, `volume_type`, `s_morning`, `s_after_issue`, `tz_end`, `end_day`, `sold_cash`, `delta_prev`, `diff`, `photo_path`, `created_by`
- `inventory.popcorn_roll_input`
  - `id`, `daily_id`, `rolls_count`, `bucket_factor`

### Поиск/создание записи
Ключ: номер телефона в формате `79781232121`.

### Поля учёта
- Телефон
- Школа
- Район
- Доп. информация

- Купленные билеты

- Рейтинг порядочности
- Фото нарушений/грязи

### Таблицы

  - `id`, `class_profile_id`, `session_at`, `tickets_count`, `rows_info`, `seats_info`, `clean_rating`, `behavior_rating`, `comment`, `created_by`
- `classes.class_evidence`
  - `id`, `visit_id`, `file_path`, `description`, `created_at`

## 8) Модуль «Чек-листы»

### Конструктор сценариев (admin/system_admin)
- Администратор создаёт чек-лист:
  - название
  - шаги
  - обязательность фото для шага
  - расписание (дата/дни недели/время)

### Выполнение сотрудником
- Открывает чек-лист
- Видит последовательность шагов
- Нажимает кнопку шага → шаг помечается ✅
- Если шаг требует фото — бот запрашивает фото

Если фото относится к разным шагам, каждое фото сохраняется как отдельное вложение, привязанное к конкретному действию.

### Напоминания
- По дате и дням недели
- Пуш в Telegram ответственным сотрудникам
- Эскалация администратору при просрочке

### Таблицы
- `checklists.template`
- `checklists.template_step`
- `checklists.assignment`
- `checklists.assignment_step_state`
- `checklists.step_attachment`

## 10) Именование файлов и поиск

Для всех загружаемых файлов (фото/документы):
`{module}_{entityCode}_{YYYYMMDD_HHMMSS}_{shortuuid}.{ext}`

Пример:
`inventory_POPCORN15_20260220_213011_a1b2c3.jpg`

Где `entityCode` — индивидуальный код сущности (сертификата, шага, отчёта и т.д.), по которому можно выполнять быстрый поиск.

## 11) Безопасность

1. Только параметризованные SQL-запросы (ORM/Prepared Statements)
2. Валидация ввода (телефон, даты, лимиты, mime-type)
3. RBAC-проверка перед каждым действием
4. Ограничение частоты запросов (антиспам)
5. Проверка и безопасное хранение файлов
6. Шифрование чувствительных данных (при необходимости)
7. Аудит-лог всех изменений

---

## 12) Логирование и аудит

Логировать:
- входы пользователей
- создание/изменение/удаление записей

- расхождения в попкорне

Аудит-таблица:
- `audit.action_log(id, actor_id, module, action, entity_type, entity_id, before, after, created_at)`

## 14) Пошаговый план внедрения

### Этап 1. Базовый каркас
- aiogram-бот, роли, авторизация, главное меню, логирование.

### Этап 2. Сертификаты
- Импорт сертификатов, поиск по коду и частичному коду, QR-распознавание.

### Этап 3. Расход попкорна
- Формы ввода, расчёты, валидация, отчётность, хранение 45 дней.

### Этап 4. Классы
- Профиль по телефону, визиты, рейтинги, фотофиксация.

### Этап 5. Чек-листы + напоминания
- Конструктор, выполнение, фото по шагам, расписание уведомлений.

### Этап 6. График
- Месячный план и персональные смены.

### Этап 7. Усиление безопасности и мониторинга
- throttling, Sentry, резервное копирование, health checks.
